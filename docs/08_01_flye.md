# De novo genome assembly with `Flye`

---

## 🎯 Learning objectives

By the end of this practical, you should be able to:

- explain why long reads are useful for de novo genome assembly
- distinguish between the main Flye input modes
- create and activate a Conda environment containing Flye
- run Flye on Oxford Nanopore long reads
- identify the main Flye output files
- inspect assembly size, contig count, coverage, and circularity
- recognize common warning signs in a long-read assembly

---

## Overview

Genome assembly reconstructs longer genomic sequences from sequencing reads. In a **de novo** assembly, reads are assembled without using a reference genome to determine their order.

`Flye` is a de novo assembler developed for long reads generated by Oxford Nanopore Technologies (ONT) and PacBio. It constructs a repeat graph, uses long reads to resolve repeated regions, and produces polished contigs.

Long reads can span repetitive regions that are difficult to resolve with short reads. For bacterial isolate genomes, a successful long-read assembly may produce:

- one circular chromosome
- one or more circular plasmids
- a small number of unresolved contigs

However, the result depends on read length, sequencing depth, DNA quality, contamination, and genome complexity.

{: .important }
> Flye requires **long reads** from ONT or PacBio. Do not use paired-end Illumina files such as `sample_1.fastq.gz` and `sample_2.fastq.gz` as Flye input. Those files require a short-read or hybrid assembler.

{: .note }
> Flye accepts FASTA or FASTQ input, either uncompressed or compressed with `gzip`. A modern ONT dataset is commonly provided as one `.fastq.gz` file containing all reads for one sample.

## 1. Create and activate the Conda environment

Move to the main workshop directory.

```bash
cd ~/2026-bioinf-ToT-EPIC-Namibia
```

Create a Conda environment containing Flye and SeqKit.

```bash
conda create -n flye \
  -c conda-forge \
  -c bioconda \
  flye seqkit \
  -y
```

Activate the environment.

```bash
conda activate flye
```

Check that both programs are available.

```bash
flye --version
seqkit version
```

If both commands print version information, the environment is ready.

```bash
cd ~/2026-bioinf-ToT-EPIC-Namibia
```

```bash
find /shared/fastq_files -maxdepth 2 -type f \
```

Identify the ONT long-read file assigned to your group or sample.

Check the link.

```bash

```

Confirm that the file exists.

```bash
ls -lh "$READS"
```

{: .warning }
> Replace `ONT_SAMPLE` with your real sample name. If the input filename does not match the value assigned to `SAMPLE`, the later commands will fail with a “file not found” error.

## 3. Inspect the long-read input

Use SeqKit to summarize the reads.

```bash
seqkit stats -a "$READS"
```

Important values include:

| Metric | Meaning |
|---|---|
| `num_seqs` | Number of reads |
| `sum_len` | Total number of sequenced bases |
| `min_len` | Length of the shortest read |
| `avg_len` | Mean read length |
| `N50` | Read length at which half of all sequenced bases are in reads of this length or longer |
| `max_len` | Length of the longest read |

For bacterial long-read assembly, useful datasets generally contain reads several kilobases long and sufficient total sequence to cover the genome many times.

A rough coverage estimate can be calculated as:

```text
coverage = total bases in reads / expected genome size
```

For example, if a bacterial genome is approximately 5.5 Mb and the reads contain 275 Mb in total:

```text
275 Mb / 5.5 Mb = approximately 50x coverage
```

{: .task }
> Inspect your input with `seqkit stats -a`. Record the read count, total number of bases, mean read length, read N50, and maximum read length.

{: .discussion }
> Would a dataset containing millions of 150 bp reads be suitable for Flye? What feature of long reads makes them more useful for resolving repeated genomic regions?

## 4. Select the correct Flye read mode

Flye requires one input mode describing the sequencing technology and expected read accuracy.

| Read type | Flye option | Typical use |
|---|---|---|
| Modern high-quality ONT reads | `--nano-hq` | R10 reads and recent high-accuracy ONT basecalling |
| Older or lower-accuracy ONT reads | `--nano-raw` | Older ONT chemistry or reads with a relatively high error rate |
| Error-corrected ONT reads | `--nano-corr` | ONT reads corrected by another program |
| PacBio HiFi reads | `--pacbio-hifi` | Highly accurate PacBio CCS/HiFi reads |
| PacBio CLR reads | `--pacbio-raw` | Older continuous long-read PacBio data |
| Error-corrected PacBio reads | `--pacbio-corr` | PacBio reads corrected by another program |

For modern ONT R10 data, use `--nano-hq`.

For R9 data basecalled with a recent high-accuracy model, Flye documentation recommends `--nano-hq` with an adjusted read-error estimate:

```bash
--nano-hq --read-error 0.05
```

{: .note }
> The correct mode depends on sequencing chemistry and basecalling, not only on the `.fastq` filename. Ask the instructor when the sequencing configuration is unknown.

## 5. Examine the Flye command

The basic command for modern ONT reads is:

```bash
flye \
  --nano-hq "$READS" \
  --out-dir output/${SAMPLE} \
  --threads 8
```

The options used here are:

| Option | Meaning |
|---|---|
| `--nano-hq` | Treat the input as high-quality ONT long reads |
| `"$READS"` | Input FASTQ file |
| `--out-dir` | Directory in which Flye writes its results |
| `--threads 8` | Use eight CPU threads |

The estimated genome size is optional in current Flye versions. For most bacterial isolate assemblies, Flye can infer the required parameters from the reads.

> What would happen if `--nano-hq` were replaced by `--pacbio-hifi` for an ONT dataset?

## 6. Run Flye

Run the assembly.

```bash
flye \
  --nano-hq "$READS" \
  --out-dir output/${SAMPLE} \
  --threads 8
```

Flye prints progress information to the terminal and also writes a log file inside the output directory.

While the assembly is running, you can examine the end of the log from another terminal.

```bash
tail -f output/${SAMPLE}/flye.log
```

Press `Ctrl+C` to stop following the log. This does **not** stop Flye when Flye is running in a different terminal.

After Flye finishes, inspect the last lines of the log.

```bash
tail -n 25 output/${SAMPLE}/flye.log
```

Look for a message indicating that the pipeline completed successfully.

{: .task }
> Run Flye on your assigned ONT sample. Do not close the terminal while the command is running.

---

## 7. Resume an interrupted run

If a Flye run was interrupted after completing at least one stage, it can often be resumed using the same input and output directory.

```bash
flye \
  --nano-hq "$READS" \
  --out-dir output/${SAMPLE} \
  --threads 8 \
  --resume
```

{: .important }
> Use `--resume` only with the same sample, read mode, and output directory. Do not resume an old output directory using a different input dataset.

## 8. Inspect the Flye output files

List the output directory.

```bash
ls -lh output/${SAMPLE}/
```

The main files are:

| File | Description |
|---|---|
| `assembly.fasta` | Final polished assembly; this is the main FASTA file used for downstream analysis |
| `assembly_info.txt` | Per-contig information, including length, coverage, circularity, repeat status, and graph path |
| `assembly_graph.gfa` | Final assembly graph in GFA format |
| `assembly_graph.gv` | Assembly graph in Graphviz format |
| `flye.log` | Detailed log of the Flye run |
| `params.json` | Parameters recorded for the run |

Flye also creates intermediate stage directories. These are useful for debugging, but `assembly.fasta` is the main assembly result.

Check that the final assembly exists and is not empty.

```bash
ls -lh output/${SAMPLE}/assembly.fasta
```

Count the assembled sequences.

```bash
grep -c '^>' output/${SAMPLE}/assembly.fasta
```

Show the FASTA headers.

```bash
grep '^>' output/${SAMPLE}/assembly.fasta
```

Summarize the assembly with SeqKit.

```bash
seqkit stats -a output/${SAMPLE}/assembly.fasta
```

{: .task }
> Record the number of contigs, total assembly length, shortest contig, longest contig, and assembly N50.

## 9. Inspect `assembly_info.txt`

Format the table for easier reading.

```bash
column -t output/${SAMPLE}/assembly_info.txt | less -S
```

Press `q` to exit `less`.

The table contains information such as:

| Column | Meaning |
|---|---|
| Sequence name | Contig identifier used in `assembly.fasta` |
| Length | Contig length in base pairs |
| Coverage | Estimated read depth for the contig |
| Circular | Whether Flye detected the contig as circular |
| Repeat | Whether the sequence is classified as repetitive |
| Multiplicity | Estimated copy number based on coverage |
| Alternative group | Relationship to possible alternative contigs |
| Graph path | Path through the Flye repeat graph |

Display only the header and contigs marked as circular.

```bash
awk 'NR == 1 || $4 == "Y"' output/${SAMPLE}/assembly_info.txt \
  | column -t
```

{: .warning }
> A circularity flag is useful evidence, but it is not by itself proof that a chromosome or plasmid is complete and correct. Assembly quality must also be assessed using coverage, assembly graphs, read mapping, contamination checks, and tools introduced in the Assembly QC practical.

## 10. Interpret the first assembly results

For a bacterial isolate, consider the following questions:

1. Is the total assembly size plausible for the expected organism?
2. Is there one large chromosome-sized contig?
3. Are smaller contigs present that could represent plasmids?
4. Are any contigs marked as circular?
5. Do contigs have similar coverage, or are there large coverage differences?
6. Is the assembly unexpectedly fragmented?
7. Is the total assembly size much larger than expected?

Possible interpretations include:

| Observation | Possible explanation |
|---|---|
| One large circular contig | Potentially complete bacterial chromosome |
| One large contig plus small circular contigs | Chromosome plus possible plasmids |
| Many short contigs | Low coverage, short reads, contamination, repeated regions, or mixed organisms |
| Assembly much larger than expected | Contamination, mixed culture, duplicated regions, or an incorrect genome-size expectation |
| Very different contig coverages | Plasmids, contamination, copy-number differences, or mixed organisms |
| No circular contigs | Genome may still be useful but unresolved repeats or insufficient read length may prevent closure |

{: .discussion }
> Compare your assembly with another group. Which assembly is more contiguous? Are differences better explained by read count, read length, coverage, or biological variation?

## 11. Optional: assemble older ONT reads

For older or lower-accuracy ONT reads, use `--nano-raw` instead of `--nano-hq`.

```bash
flye \
  --nano-raw "$READS" \
  --out-dir output/${SAMPLE}_nano_raw \
  --threads 8
```

Do not run both modes merely to choose the assembly with the fewest contigs. Select the mode based on the sequencing chemistry and expected read accuracy.

---

## 12. Optional: metagenome assembly

Flye has a metagenome mode for datasets containing organisms at uneven abundance.

```bash
flye \
  --nano-hq "$READS" \
  --meta \
  --out-dir output/${SAMPLE}_metaflye \
  --threads 8
```

{: .important }
> Do not automatically use `--meta` for a cultured bacterial isolate. Standard Flye mode assumes relatively uniform genome coverage and is generally more appropriate for a single isolate. Use `--meta` for a true mixed-community dataset or when uneven coverage is expected.

---

## 13. Optional: reduce memory use for a high-coverage dataset

For a very large or very high-coverage dataset, Flye can use a subset of the longest reads during the initial assembly stage.

This requires both `--asm-coverage` and `--genome-size`.

For an organism with an expected genome size of approximately 5.5 Mb:

```bash
flye \
  --nano-hq "$READS" \
  --genome-size 5.5m \
  --asm-coverage 40 \
  --out-dir output/${SAMPLE}_cov40 \
  --threads 8
```

All reads are still used during later stages such as repeat resolution and polishing.

{: .note }
> This option is normally unnecessary for a modest bacterial dataset. Use it when memory consumption is a genuine problem or when instructed to do so.

## 14. Common problems

### Flye reports that the input file does not exist

Check the variable and symlink.

```bash
echo "$READS"
ls -lh "$READS"
```

Confirm that `SAMPLE` matches the real filename.

### The output directory already contains a previous run

Use a new output directory or resume the existing run deliberately.

```bash
flye \
  --nano-hq "$READS" \
  --out-dir output/${SAMPLE}_run2 \
  --threads 8
```

### The assembly contains many contigs

Check:

- whether the input really contains long reads
- read N50 and maximum read length
- total sequencing yield and estimated coverage
- whether the sample is contaminated or mixed
- whether the correct Flye read mode was selected
- the warnings and errors in `flye.log`

### Flye is killed during the run

A process terminated with `Killed` or exit code `137` was commonly stopped because the system ran out of memory.

Possible actions include:

- run on a machine with more RAM
- avoid running several large jobs simultaneously
- use `--asm-coverage` together with a realistic `--genome-size`
- ask the instructor to check the server workload

### The input consists of paired-end Illumina reads

Do not concatenate the paired files and pass them to Flye. Flye is a long-read assembler. Use a short-read assembler or a hybrid assembly workflow instead.
